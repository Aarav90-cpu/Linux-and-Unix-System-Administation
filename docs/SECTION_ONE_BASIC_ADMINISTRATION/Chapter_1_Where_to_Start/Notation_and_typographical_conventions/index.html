1.5 Notation and Typographical Conventions

To keep things from devolving into total chaos, we use a few strict rules when showing you commands and code.

Fonts and Formatting

  • Bold text: Filenames, commands, and literal arguments you need to type exactly as shown.
  • Italics: Placeholders. Don't actually type cp file directory. Replace them with the real things!
  • Code font: Used for excerpts from config files and terminal sessions.

Sometimes, we drop little hints inside terminal blocks using the bash comment character (#) followed by italics:

$ grep Bob /pub/phonelist # Look up Bob's phone number
Bob Knowles 555-2834
Bob Smith 555-2311

Terminal Prompts

The prompt tells you who you need to be to run the command:

  • $ means run it as a normal, unprivileged user.
  • # means you need to be root (the superuser).

If a command only works on a specific distro, we'll prefix the prompt with the distro name so you don't break your server trying to use Debian tools on Red Hat:

$ sudo su - root # Become root
# passwd # Change root's password
debian# dpkg -l # List installed packages on Debian and Ubuntu

Syntax Cheatsheet

We use standard man page conventions when showing how a command is structured:

  • [ ] (Square brackets): The stuff inside is optional.
  • ... (Ellipsis): You can repeat the previous item.
  • { | } (Curly braces with vertical bars): Pick exactly one of the items separated by the bars.

So, bork [ -x ] { on | off } filename ... means you could run bork -x off /etc/passwd /etc/smartd.conf.

Globbing (Wildcards)

Shell pattern matching makes life easier:

  • * (Star): Matches zero or more characters.
  • ? (Question mark): Matches exactly one character.
  • ~ (Tilde): Your home directory. (And ~user is that specific user's home directory).

P.S. When we quote things, we put punctuation outside the quotes if it belongs outside. We're sysadmins, not grammar teachers, and precision is more important than U.S. English rules.
